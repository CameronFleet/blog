`CLAUDE.md` is config. We ship it like a comment. That's the problem.

I watched a developer on my team add a single sentence to our shared `CLAUDE.md` — clarifying how to handle an edge case — and silently break the behavior dozens of people relied on every day. No tests caught it. No CI flagged it. The first signal was someone asking why Claude was behaving differently.

If they'd made the same change to a service config file, we'd have demanded a test. But `CLAUDE.md`? We trusted the vibes.

This post is me thinking through why that's not good enough, what the ecosystem has built to fix it, and what a proper CI pipeline actually looks like.

## What CLAUDE.md actually is

`CLAUDE.md` (and `AGENTS.md`) is runtime configuration for a probabilistic system. It gets injected into the system prompt. It defines constraints, tone, tool behavior, conventions. Everything that shapes how an agent interprets and responds to every request.

The important difference from normal config: a broken value in a service config usually produces an immediate, deterministic error. A broken instruction in `CLAUDE.md` produces subtly wrong behavior that varies by prompt, context, and model temperature. There's no stack trace. There's no `500`. There's just drift.

Anthropic describes `CLAUDE.md` as something you check into version control alongside your code. The [AGENTS.md community spec](https://agents.md/) formalizes this. GitHub's analysis of over 2,500 repos found these files are updated roughly every 22 hours — many hands, small increments, constantly.

That cadence sounds healthy. Without tests, it's a slow-moving accident.

## The current state: hope and a reference prompt

At my team, the informal process is: when someone wants to change `CLAUDE.md`, run a set of reference prompts before and after and look at the outputs. Manual. Subjective. Skipped when people are busy.

Better than nothing. Still has obvious problems:

- **Coverage is ad hoc.** The reference prompts reflect whatever the author happened to think of, not a systematic representation of how Claude is actually used.
- **Comparison is subjective.** Two engineers looking at the same before/after will often disagree about whether it's a regression.
- **Process is opt-in.** Small "obvious" changes skip it entirely.
- **Nondeterminism obscures signal.** Even identical inputs produce different outputs. How do you know if the difference is from your change or just natural variation?

These aren't unique to `CLAUDE.md` — they're the general problem of prompt regression testing. The good news is that field has been moving fast.

## What the ecosystem has built

The broader problem of "test your LLM configuration changes" is well understood. Nothing is purpose-built specifically for `CLAUDE.md` diffs, but the pieces are there.

### Promptfoo

[Promptfoo](https://github.com/promptfoo/promptfoo) is the most complete open-source solution. You write YAML test configs that define your system prompt (which can include your `CLAUDE.md`), test inputs, and assertions over outputs. Assertions can be deterministic (does the output contain this string? Is it valid JSON?) or semantic (does an LLM-as-judge score this above 0.8?).

It has a native [Claude Agent SDK provider](https://www.promptfoo.dev/docs/providers/claude-agent-sdk/), a [GitHub Action](https://github.com/promptfoo/promptfoo-action) that posts results as PR comments, and can fail the build on regressions. If I were starting from scratch, I'd start here.

```yaml
# promptfooconfig.yaml
prompts:
  - file://CLAUDE.md

providers:
  - id: anthropic:claude-sonnet-4-6

tests:
  - vars:
      task: "Write a SQL migration to add a nullable column"
    assert:
      - type: llm-rubric
        value: "Response includes a rollback strategy and does not use ALTER TABLE without a transaction"
      - type: not-contains
        value: "DROP TABLE"
```

### Agent CI

[Agent CI](https://agent-ci.com) is newer and closer to what I actually want: a GitHub App that auto-discovers agent configurations and runs behavioral evals on every PR. It explicitly models variance across multiple runs — running the same eval several times and flagging inconsistency. That's the right instinct for nondeterministic systems.

### DeepEval

[DeepEval](https://deepeval.com) takes a pytest-first approach. LLM tests that look exactly like unit tests, with built-in metrics for hallucination, bias, relevancy, and more. If your team is already invested in pytest and wants to treat prompt changes with the same discipline as code changes, this is the natural fit.

### The LLM-as-judge pattern

All of these tools share a core approach for handling nondeterminism: use another LLM to evaluate the output. Instead of asserting that the output equals a specific string, you ask a judge model to score the output against a rubric.

The key practices for making this reliable:
- Constrain the judge to binary or narrow outputs (pass/fail, 1–5) — not open-ended assessments
- Run the same eval 3–5 times and treat inconsistent judge results as a signal that your rubric is underspecified
- Use a lower temperature for judge calls (0.2–0.3)
- Build rubrics as explicit decision trees, not open-ended prompts

Golden datasets — curated inputs with expected *properties*, not exact outputs — are the foundation. Golden datasets for CI gates, random sampling for catching emergent failures in production.

## What a proper CI pipeline looks like

Here's the architecture I'd reach for:

```
PR opens with CLAUDE.md change
        │
        ▼
┌─────────────────────────────┐
│   promptfoo-action runs     │
│   eval suite in CI          │
└──────────────┬──────────────┘
               │
       ┌───────┴────────┐
       ▼                ▼
  Deterministic    LLM-as-Judge
  assertions       rubric scores
  (format,         (behavior,
  safety rules,    intent,
  must-not-do)     quality)
       │                │
       └───────┬────────┘
               ▼
       Score vs. threshold
               │
       ┌───────┴────────┐
       ▼                ▼
    PASS              FAIL
  merge allowed    PR blocked +
                  results posted
                  as PR comment
```

**The eval suite lives in version control alongside `CLAUDE.md`.** Changes to the file and changes to the tests travel together in the same PR. That's the critical discipline — treating the test suite as the specification of what `CLAUDE.md` is supposed to do.

Two layers:

1. **Hard assertions.** Things that should never happen regardless of context. Never suggest `DROP TABLE` without confirmation. Always include a rollback. Never expose API keys. Deterministic, cheap to run.

2. **Behavioral rubrics.** Evaluated by an LLM judge. "Does this response follow our convention for commit messages?" "Does this advice account for our blockchain-specific constraints?" These are harder, but they're the important ones.

## The nondeterminism problem is real but manageable

The most common objection: how do you write reliable tests for a probabilistic system?

A few approaches that help:

**Run N times, require M passes.** Don't assert that a single run passes. Run the eval 5 times and require 4/5 to pass. Genuine regressions fail consistently. Random variation passes most of the time.

**Track score distributions, not just pass/fail.** A healthy `CLAUDE.md` produces outputs that score above your threshold consistently. If your change shifts the distribution down — even if individual runs still technically pass — that's a signal worth investigating.

**Pairwise comparison is underrated.** For catching regressions specifically, running the same inputs against old and new `CLAUDE.md` and asking a judge to compare them directly is more sensitive than scoring each independently. The judge doesn't need to know what "good" looks like in the abstract — just which output is better.

**Update the model separately from the configuration.** Many silent prompt regressions come not from changes you made but from Anthropic updating the underlying model. Running your eval suite on a schedule — not just on PRs — catches these.

## The honest gap

The tooling exists. What doesn't exist is a widely-adopted, standardised workflow specifically for `CLAUDE.md` testing. The promptfoo + GitHub Actions path is the closest thing to turnkey, but you still have to write the evals, which requires thinking carefully about what your `CLAUDE.md` is actually supposed to accomplish.

That last part is actually the most valuable forcing function. The reason most `CLAUDE.md` files drift is that nobody ever wrote down what they were supposed to do in testable terms. Writing evals forces you to do that.

The file that tells your AI agent how to behave is at minimum as important as your `.eslintrc`. Probably more so. We should be treating it that way — and the tooling to do that now exists. There's no excuse left.
